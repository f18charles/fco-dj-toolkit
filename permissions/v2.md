# Antigravity CLI Prompt — `permissions` module v2

---

## Before anything else

You are upgrading an existing, fully tested, production-grade v1 module.
The rules are stricter than a greenfield build:

1. **Do not break v1.** Every public method, class, and import path that
   existed in v1 must continue to work unchanged. v2 adds — it does not
   remove or rename anything v1 callers depend on.
2. **Do not regenerate v1 files from scratch.** Edit existing files
   surgically. Add new files for new features. The only exception is
   `migrations/` — new migrations are additive, never replacements.
3. **Run the full v1 test suite before writing a single line of v2 code.**
   If it passes, proceed. If it doesn't, fix v1 regressions first before
   adding anything new. Log the outcome to `agy.md`.
4. **agy.md rule:** After completing every individual feature, append
   exactly one line:
   ```
   [DONE] <feature_name> — <one sentence max>
   ```
   First entry before any code:
   ```
   [V2 START] permissions v2 — v1 suite passed, beginning v2 build
   ```

## What v2 adds (complete list, nothing else)

1. Object-level permissions (`ObjectPermission` model + `ObjectPermissionService`)
2. Generic scope system (forward-compatible foundation for `organizations` v2)
3. `ScopedUserRole` model (role assignment scoped to an arbitrary scope object)
4. `PermissionGroup` model (display grouping for admin UI — no runtime effect)
5. Redis-backed cross-request permission cache (opt-in, falls back to v1 in-memory cache)
6. Template tags (`{% has_permission %}`, `{% has_role %}`, `{% has_object_permission %}`)
7. `clean_expired_assignments` management command
8. DRF integration tests (end-to-end request/response cycles, not just unit tests)
9. Audit signal receivers example module (`permissions/audit.py`)
10. `PermissionService.has_permission()` extended with optional `scope` and `obj` arguments
    — existing callers with no `scope`/`obj` are completely unaffected

---

## Dependency note: `organizations` module

The `organizations` module does not exist yet in fco-dj-kit. v2 must NOT
import from `organizations` anywhere. Instead, the scope system is built
generically: a "scope" is any model instance (identified by
`content_type + object_id`, a GenericForeignKey). When `organizations`
is built later, it will plug in by passing an `Organization` instance
as the `scope` argument — no changes to `permissions` will be required.

## New models — add to `models.py`

### `ObjectPermission`

Grants a specific permission to a specific user on a specific model instance.
Uses `GenericForeignKey` so it works with any model in any app.

```
user          ForeignKey(AUTH_USER_MODEL, on_delete=CASCADE,
              related_name="object_permissions")

permission    ForeignKey(Permission, on_delete=CASCADE,
              related_name="object_permissions")

content_type  ForeignKey(ContentType, on_delete=CASCADE)
              The model class the object belongs to.

object_id     CharField(max_length=255)
              String representation of the target object's PK.
              CharField (not PositiveIntegerField) because PKs may be
              UUIDs, strings, or integers depending on the model.

content_object  GenericForeignKey("content_type", "object_id")
                The actual target object. Not stored — resolved at runtime.

granted_by    ForeignKey(AUTH_USER_MODEL, null=True, blank=True,
              on_delete=SET_NULL, related_name="granted_object_permissions")

expires_at    DateTimeField(null=True, blank=True)
```

Meta:
```
unique_together = [("user", "permission", "content_type", "object_id")]
indexes = [
    Index(fields=["content_type", "object_id"]),
    Index(fields=["user", "content_type"]),
]
```

Inherits from `UUIDModel`, `TimestampedModel`.

### `ScopedUserRole`

A role assignment scoped to a specific object (e.g. an organization, a
project, a workspace). When `organizations` exists, it passes an
`Organization` instance as the scope. The permissions module does not
know or care what the scope object is — it's just a GenericForeignKey.

```
user          ForeignKey(AUTH_USER_MODEL, on_delete=CASCADE,
              related_name="scoped_user_roles")

role          ForeignKey(Role, on_delete=CASCADE,
              related_name="scoped_user_roles")

content_type  ForeignKey(ContentType, on_delete=CASCADE)
object_id     CharField(max_length=255)
scope         GenericForeignKey("content_type", "object_id")
              The object this role assignment is scoped to.

granted_by    ForeignKey(AUTH_USER_MODEL, null=True, blank=True,
              on_delete=SET_NULL, related_name="granted_scoped_user_roles")

expires_at    DateTimeField(null=True, blank=True)
```

Meta:
```
unique_together = [("user", "role", "content_type", "object_id")]
```

Inherits from `UUIDModel`, `TimestampedModel`.

## Cache layer — `cache.py` (new file)

v1 cached the resolved permission set as `user._permissions_cache` (an
attribute on the user object, in-memory, per-request only). v2 introduces
an optional Redis-backed cache that persists across requests and processes,
so a permission check for the same user on a second request doesn't re-query
the database.

### Design rules

- Falls back to v1 in-memory behaviour if `PERMISSIONS_CACHE_BACKEND` is
  not set in settings. The v1 code path must remain intact.
- Uses Django's cache framework (`django.core.cache.caches`) — not a direct
  Redis client — so any Django cache backend works (Redis, Memcached, local
  memory for tests).
- Cache key format: `permissions:user:{user_pk}:all_perms`
- TTL: configurable via `PERMISSIONS_CACHE_TTL` (default: 300 seconds).
- Invalidation: on `UserRole`, `UserPermission`, `ScopedUserRole`,
  `ObjectPermission` save/delete, the cache key for the affected user is
  deleted immediately (not waited on TTL).
- Object-level and scoped permissions are NOT cached (too many possible
  combinations). Only the global permission set is cached.

```python
# cache.py public interface

def get_user_permissions_from_cache(user) -> set[str] | None:
    """
    Return the cached permission set for user, or None if not cached.
    Checks Redis first (if configured), then falls back to in-memory.
    """

def set_user_permissions_cache(user, permissions: set[str]) -> None:
    """Store the resolved permission set. Writes to Redis if configured."""

def invalidate_user_permissions_cache(user) -> None:
    """
    Delete all cached permission data for user.
    Called by signal receivers on UserRole / UserPermission changes.
    Must handle the case where the user has no cached data gracefully.
    """
```

### Settings

```python
# Use a specific Django cache alias for permissions.
# If not set, uses in-memory per-request caching (v1 behaviour).
PERMISSIONS_CACHE_BACKEND = "permissions"   # must match a key in CACHE settings

# How long to cache the resolved permission set (seconds).
PERMISSIONS_CACHE_TTL = 300
```

Document the recommended cache config in `docs/caching.md`:

```python
# Example: separate Redis cache for permissions
CACHES = {
    "default": { ... },
    "permissions": {
        "BACKEND": "django.core.cache.backends.redis.RedisCache",
        "LOCATION": "redis://127.0.0.1:6379/1",
        "TIMEOUT": 300,
        "KEY_PREFIX": "fco_permissions",
    }
}
```

## Extended `services.py`

### `PermissionService.has_permission` — extended signature

```python
@staticmethod
def has_permission(
    *,
    user,
    codename: str,
    obj=None,          # NEW — if provided, also checks ObjectPermission
    scope=None,        # NEW — if provided, also checks ScopedUserRole permissions
) -> bool:
```

Resolution order with v2 arguments:

1. Superuser bypass (unchanged from v1, checked first regardless of obj/scope)
2. If `obj` is provided: check active `ObjectPermission` for `(user, codename, obj)` → return True if found
3. If `scope` is provided: check active `ScopedUserRole` assignments for `(user, scope)`, for each role check its permissions + parent role permissions + wildcards
4. Global check (unchanged v1 logic): direct grants → roles → inherited → wildcards

Existing callers passing neither `obj` nor `scope` follow the exact same
code path as v1. No behaviour change, no performance change.

### New: `ObjectPermissionService`

New service class. All methods `@staticmethod`, keyword-only.

```python
@staticmethod
def grant(
    *, user, codename: str, obj, granted_by=None, expires_at=None
) -> ObjectPermission:
    """
    Grant user permission `codename` on the specific object `obj`.

    `obj` can be any model instance. The content_type and object_id are
    resolved automatically.

    Raises:
        PermissionNotFoundError: if codename doesn't exist in Permission table.
        ObjectPermissionAlreadyGrantedError: if already granted.
    """

@staticmethod
def revoke(*, user, codename: str, obj) -> None:
    """
    Raises:
        PermissionNotFoundError: if codename doesn't exist.
        ObjectPermissionNotFoundError: if grant doesn't exist.
    """

@staticmethod
def has_object_permission(*, user, codename: str, obj) -> bool:
    """
    Check if user has `codename` on the specific object `obj`.

    Does NOT fall through to global permissions — this is a pure
    object-level check. If you want "global OR object-level",
    use PermissionService.has_permission(user=user, codename=codename, obj=obj).

    Never raises. Returns False for any error condition.
    """

@staticmethod
def get_users_with_permission_on_object(codename: str, obj) -> QuerySet:
    """
    Return all users who have `codename` on the specific object `obj`.
    Active grants only (respects expires_at).
    """

@staticmethod
def get_objects_user_can_access(user, codename: str, model_class) -> QuerySet:
    """
    Return a queryset of `model_class` instances that `user` has
    `codename` permission on via ObjectPermission.

    Note: does NOT include objects the user can access via a global
    permission — this is purely the object-level grant set.
    Documents this limitation in its docstring.
    """
```

```python
@staticmethod
def assign_scoped_role(
    *, user, role_name: str, scope, granted_by=None, expires_at=None
) -> ScopedUserRole:
    """
    Assign a role to a user scoped to `scope` (any model instance).

    Raises:

@staticmethod
def revoke_scoped_role(*, user, role_name: str, scope) -> None:

    Raises:
        RoleNotFoundError, PermissionsModuleError if assignment not found.
    """

@staticmethod

@staticmethod

    Does NOT include global permissions — scoped and global are separate.
    """
```

## Template tags — `templatetags/permissions_tags.py`

```python
{% load permissions_tags %}

{% has_permission request.user "posts.edit" as can_edit %}
{% if can_edit %}...{% endif %}

{% has_role request.user "editor" as is_editor %}
{% if is_editor %}...{% endif %}

{% has_object_permission request.user "posts.edit" post as can_edit_post %}
{% if can_edit_post %}...{% endif %}

{% has_any_permission request.user "posts.edit,posts.publish" as can_do_something %}
{% if can_do_something %}...{% endif %}
```

Implementation rules:
- All tags are assignment tags (`as <varname>`) — they never render directly,
  they only set a context variable. This keeps logic out of templates.
- Each tag calls its corresponding `PermissionService` or
  `ObjectPermissionService` method — no direct DB queries in tag code.
- Tags fail silently (return False) rather than raising — a broken
  permission check should not crash a page render.
- Requires `django.template.context_processors.request` in `TEMPLATES`
  settings. Document this requirement in README.

## Management command — `management/commands/clean_expired_assignments.py`

```
python manage.py clean_expired_assignments [--dry-run] [--older-than DAYS]
```

- Without `--dry-run`: deletes all expired `UserRole`, `UserPermission`,
  `ScopedUserRole`, and `ObjectPermission` rows where `expires_at <= now()`.
- `--older-than DAYS` (default: 0): only delete rows that expired more than
  N days ago, giving a grace window before permanent deletion.
- `--dry-run`: prints counts of what would be deleted without deleting anything.
- Logs a summary line at `INFO` level after each model is cleaned.
- Returns a non-zero exit code if any deletion raises an unexpected exception,
  so a cron job / scheduled task can detect failures.

```python

Dry run mode — no records will be deleted.
UserRole: 12 expired records would be deleted (older than 0 days)
UserPermission: 3 expired records would be deleted
ScopedUserRole: 0 expired records would be deleted
ObjectPermission: 7 expired records would be deleted
Total: 22 records would be deleted.
```

## Extended `drf/permissions.py`

Add these classes. Do not modify v1 DRF classes.

```python
class HasObjectPermission(BasePermission):
    """
    DRF object-level permission class.
    Calls ObjectPermissionService.has_object_permission() in has_object_permission().

    Usage:
        class PostDetailView(RetrieveUpdateAPIView):
            permission_classes = [IsAuthenticated, HasObjectPermission]
            required_permission = "posts.edit"

    has_object_permission() is called automatically by DRF's APIView
    when check_object_permissions() is invoked (e.g. via get_object()).
    """
    required_permission: str = ""

    def has_permission(self, request, view) -> bool:
        return request.user and request.user.is_authenticated

    def has_object_permission(self, request, view, obj) -> bool: ...


class HasScopedPermission(BasePermission):
    """
    Checks PermissionService.has_permission with a scope resolved from the view.
    View must implement get_permission_scope() -> model instance.

    Usage:
        class ProjectView(APIView):
            permission_classes = [IsAuthenticated, HasScopedPermission]
            required_permission = "projects.edit"

            def get_permission_scope(self):
                return get_object_or_404(Organization, pk=self.kwargs["org_pk"])
    """
    required_permission: str = ""

    def has_permission(self, request, view) -> bool: ...
```
